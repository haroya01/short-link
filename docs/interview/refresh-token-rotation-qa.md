# 면접 문답 — 리프레시 토큰 로테이션

> 소재: `/refresh` 엔드포인트의 토큰 로테이션 + 재사용 감지.
> 표면은 "인증"이지만 실제로는 **분산 저장소의 check-then-act 경쟁**과 **보안 표준 vs 소비자 UX 트레이드오프** 두 축이 만난다.

관련 코드:
- `user/application/write/AuthService.java` (`refresh()` — 3갈래 분기)
- `user/infrastructure/RedisRefreshTokenStore.java`

---

## 배경 — 로테이션 (one-time use) 3갈래

리프레시 토큰마다 고유 `jti`, Redis에 `refresh:{userId}:{jti}` 저장. `/refresh` 요청 시:

1. **jti가 Redis에 있음** → 살아있는 세션 → 키 `delete`(1회용 소모) + "방금 로테이션됨" 유예 마커 남기고 → **새 쌍 발급**.
2. **jti 없는데 유예 마커 있음** → 다른 탭/서브도메인의 선의의 재생 → 다른 세션 안 죽이고 **재발급**.
3. **jti 없고 유예도 지남** → **이 토큰 하나만** 거절. (다른 세션 유지 — RFC 6819의 "패밀리 전체 폐기"를 일부러 안 함.)

---

## Q1. 진짜 동시(simultaneous) 경쟁

### 질문
두 서브도메인(`blog.kurl.me` + `kurl.me`)이 **같은 jti**를 들고 **완전히 동시에** `/refresh`를 쏜다. 무슨 일이 벌어지나? "1회용" 보장은 지켜지나?

### 모범답안 — 원래 토큰 하나에서 유효 세션 2개
- 두 요청 다 `exists` 통과 (누구도 아직 `delete` 안 한 시점) → **둘 다 1번 갈래**.
- 각자 `delete` → `markRotated` → `issue` → **독립된 새 쌍 2개** 발급 → **one-time use 위반**.
- **유예 마커는 여기서 안 도와준다:** 마커는 **순차(sequential)** 재생만 잡는다(A가 delete 끝낸 *뒤* B 도착 → 2번 갈래). **동시**엔 둘 다 delete 전에 exists 통과 → 둘 다 1번 → 마커 조회조차 안 됨.

### 위험한가 — "누가 쐈나"가 기준
- **둘 다 주인**(두 탭) → benign, 둘 다 본인 세션.
- **하나가 도둑** → 공격자·주인 **둘 다** 유효 세션, **아무도 도난 플래그 안 됨** → 재사용 감지가 이 창에서 조용히 실패. (단 새 권한을 주는 건 아니고 **탐지 회피**가 진짜 대가.)

### 고치는 법 — check-then-act를 원자적 연산으로
`exists`가 아니라 **`delete`의 반환값(지운 키 수)**으로 게이트. `DEL`이 `1`을 돌려준 요청만 진짜 로테이터, `0`을 받은 패자는 유예/재발급 갈래로 흐름. (`GETDEL` / Lua 스크립트 동일 원리.)
→ **Q(리다이렉트)의 원자적 `UPDATE ... WHERE`와 같은 교훈: 경쟁을 저장소의 원자적 연산에 접으면 공짜로 사라진다.**

## Q2. 3번 갈래 — RFC 6819를 일부러 어긴 부분

### 질문
표준은 "재사용 감지 = 유저 토큰 패밀리 전체 폐기"(재사용 = 도난 신호). 이 코드는 일부러 안 하고 그 토큰 하나만 거절한다. 이 결정을 공격/방어하고, **실제 도난이 이 방식을 빠져나가는 구체적 시나리오**를 대라.

### 모범답안

**교과서가 패밀리를 죽이는 이유 — 트립와이어:**
정상 클라는 항상 최신 토큰으로 전진하고 옛 토큰을 절대 재생 안 함. 재생 = 두 주체가 같은 토큰 = 도난. 성질: **누가 두 번째로 쓰든 두 번째 사용은 반드시 탐지** → 두 체인이 갈라지는 순간 누군가 죽은 토큰을 재생하고, 그때 계보를 태운다. 공격자의 장기 병렬 세션을 막는 유일한 장치.

**kurl이 내준 것 — 도난이 투명해지는 시나리오:**
1. 공격자가 살아있는 T 탈취.
2. 공격자가 **먼저** `/refresh(T)` → 1번 갈래 → T 삭제, 공격자가 자기 소유 새 쌍 T_a → **자가재생 계보 확보**.
3. 주인이 뒤늦게 T로 `/refresh(T)` → jti 없음·유예 지남 → **3번 갈래 → "이 토큰만 거절."** 교과서적 도난 신호를 보고도 어깨 으쓱.
4. 공격자는 T_a→T_a'→… 무한 로테이션. kurl은 주인의 거절과 공격자의 계보를 영영 연결 못 함 → **도난 invisible.**

**방어(스틸맨) — 이게 진짜 이유:**
"재사용 = 패밀리 폐기"는 멀티탭·멀티서브도메인·불안정 네트워크 소비자 제품에선 오탐 폭탄. 선의의 재생 원인: 두 서브도메인 동시 리프레시 / 응답 유실 후 재시도 / 잠자던 탭 / 백그라운드 복귀 모바일. 전부에서 교과서 응답 = **죄 없는 유저를 전 기기 로그아웃** = "자리 비우면 로그아웃" 신고. 상시 터짐.

**진짜 정답 — 이분법이 함정:**
- 유예 창 안 재생 → benign 재발급 (2번, OK)
- 유예 **지난** 재생 → **보안 이벤트로 기록**하되 전 세션은 안 죽임. 리스크 신호(새 IP/기기 지문·반복)면 **그 계보만** 폐기 / step-up / 알림. 트립와이어를 **등급제 알람**으로.

**구조적 진짜 원인:**
토큰이 `refresh:{userId}:{jti}`로 **납작하게** 저장 → **계보(family) 개념 부재** → 벌크 옵션이 `deleteAllForUser`(전 세션 몰살) 하나뿐 → "한 개" 아니면 "전부"밖에 못 고름. 외과적 절제가 **표현 불가능**했던 게 근본. 처방 = 리프레시 토큰에 device/family id 부여해 계보 추적.

### 판정
표준의 무딘 "재사용 = 전 폐기"가 소비자 제품엔 트리거-해피란 걸 **정확히** 짚었지만, 반동으로 **도난 신호를 통째로 버렸다.** 유예 창은 benign을 처리하나 그 너머는 전부 benign 취급 = **진짜 도둑이 사는 곳.** 근본은 UX가 아니라 계보 개념의 부재.

---

## 두 라운드 공통 교훈
- **check-then-act(exists→delete, SELECT→UPDATE)는 저장소의 단일 원자적 연산으로 접어라.** 리다이렉트·리프레시 양쪽의 핵심.
- **보안 표준을 어길 땐 "왜 어겼나 + 대신 무엇으로 위험을 흡수하나 + 언제 다시 무장하나"를 준비하라.** "몰랐다"와 "알고 골랐다"는 하늘과 땅.
